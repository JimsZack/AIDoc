# 软考系统架构设计师综合知识高分指南（保姆级终极背诵版）

> 适用科目：软考系统架构设计师综合知识（科目代码：51）<br>
> 考试形式：75 道单项选择题，每题 1 分，满分 75 分，45 分合格<br>
> 使用方法：优先背关键词、口诀、易错点；做题时看到题干关键词，直接对号入座

---

## 一、考试整体策略

### 1. 综合知识到底考什么

综合知识不是考你把某个知识点讲得多深，而是考你：

- 能不能快速识别概念
- 能不能区分相似名词
- 能不能在场景题里判断“这题在问什么”
- 能不能避开命题人设置的陷阱

### 2. 高分复习顺序

建议按下面顺序复习，效率最高：

1. **架构设计**：占比高，必须拿下
2. **软件工程与项目管理**：规则性强，容易得分
3. **安全与标准化**：记忆型强，适合突击
4. **数据库、操作系统、网络基础**：高频基础题
5. **UML、4+1 视图、可靠性、知识产权**：高频必考补充点
6. **新技术趋势**：防止被新名词吓住

### 3. 考场答题原则

- **先抓关键词，再判断知识点**
- **先排明显错误，再比较相近选项**
- **看到“最适合”“最主要”“最能体现”时，要选题干核心诉求最强的那个**
- **不要凭日常经验答题，要按教材定义答题**

## 二、架构设计模块（核心中的核心）

> 记忆总原则：**架构风格看结构，质量属性看目标，设计模式看意图，架构评估看方法。**

### 1. 常见架构风格对比

| 架构风格 | 核心特征 | 典型应用场景 | 关键词识别 |
|---|---|---|---|
| 分层架构 | 按职责分层，层间单向依赖 | Web 应用、ERP、管理系统 | 表现层、业务层、数据层、分层调用 |
| C/S 架构 | 客户端发请求，服务端集中处理 | 数据库系统、ATM、内部专用系统 | 客户端/服务器、集中式、专用客户端 |
| B/S 架构 | 浏览器作为统一客户端 | 网站、OA、在线系统 | 浏览器访问、零安装、统一入口 |
| 微服务架构 | 服务自治、独立部署、轻量通信 | 电商、云原生、大型互联网系统 | API 网关、服务注册、熔断、限流 |
| 事件驱动架构 | 基于事件触发，异步解耦 | 实时交易、消息通知、IoT | 发布/订阅、事件总线、异步 |
| 管道-过滤器 | 数据顺序流经多个处理环节 | 编译器、ETL、日志处理 | 数据流、过滤器、顺序变换 |
| 黑板架构 | 多个知识源共享黑板协作求解 | 专家系统、语音识别、诊断系统 | 黑板、知识源、推理 |
| 虚拟机架构 | 通过解释器/规则引擎执行抽象指令 | 规则系统、脚本引擎 | 虚拟机、解释执行、规则 |

#### 高频辨析

- **微服务**是**架构风格**
- **分布式**是**部署形态**
- **消息队列**是技术手段，**不是**架构风格本身
- **分层架构**关注职责拆分，**管道-过滤器**关注数据流转
- **黑板架构**适合“没有固定求解路径”的问题

### 2. 软件架构“4+1”视图模型（高频必考）

> 口诀：**逻辑看功能，开发看代码，进程看并发，物理看部署，场景串全局。**

| 视图 | 关注点 | 面向人群 | 常见图 |
|---|---|---|---|
| 逻辑视图 | 系统提供什么功能 | 用户、分析设计人员 | 类图、状态图 |
| 开发视图 | 模块、包、组件如何组织 | 程序员 | 组件图、包图 |
| 进程视图 | 并发、同步、性能、通信 | 集成人员、性能工程师 | 顺序图、活动图 |
| 物理视图 | 软件如何部署到硬件节点 | 运维、网络、硬件工程师 | 部署图 |
| 场景视图（+1） | 用例驱动，串联前四个视图 | 所有人 | 用例图 |

#### 易错点

- **开发视图**不是开发流程，而是代码模块组织关系
- **进程视图**不是进程管理理论，而是系统运行时的并发与通信
- **物理视图**重点是“部署到哪些机器/节点”
- **场景视图**本质上就是通过用例验证架构

### 3. 质量属性识别

| 质量属性 | 题干关键词 | 常见实现手段 |
|---|---|---|
| 性能 | 响应时间、吞吐量、并发、时延 | 缓存、索引、负载均衡、异步 |
| 可用性 | 冗余、故障恢复、自动切换、不中断服务 | 集群、心跳检测、主备切换 |
| 安全性 | 身份认证、授权、加密、防篡改 | TLS、数字签名、ACL、审计 |
| 可修改性 | 扩展、易改、配置化、插件化 | 解耦、接口隔离、模块化 |
| 可测试性 | Mock、替身、日志追踪、可观测 | 依赖注入、接口分层、测试桩 |
| 可伸缩性 | 横向扩容、弹性伸缩 | 无状态服务、分片、容器编排 |
| 互操作性 | 异构系统对接、协议兼容 | REST、JSON、XML、标准接口 |

#### 经典陷阱

- **负载均衡**既可能考性能，也可能考可用性，要看题干主诉求
- **日志**不只服务调试，也常用于**可测试性、可审计性、安全性**
- **模块化**主要提升**可修改性**

### 4. 常见设计模式辨析

#### 4.1 创建型模式

| 模式 | 用途 | 关键词 |
|---|---|---|
| 单例模式 | 保证全局只有一个实例 | 唯一、全局访问、懒加载 |
| 工厂方法 | 把对象创建延迟到子类/工厂 | 创建解耦、统一接口 |
| 抽象工厂 | 创建一族相关对象 | 产品族、多个关联对象 |
| 建造者 | 分步骤构建复杂对象 | 分步创建、复杂对象 |
| 原型 | 通过复制已有对象创建新对象 | 克隆、拷贝 |

#### 4.2 结构型模式

| 模式 | 用途 | 关键词 |
|---|---|---|
| 适配器 | 转换不兼容接口 | 兼容旧接口、包装 |
| 桥接 | 抽象与实现分离 | 多维变化分离 |
| 组合 | 树形结构统一处理 | 部分-整体、树结构 |
| 装饰 | 动态附加职责 | 动态增强、包装扩展 |
| 外观 | 对外提供统一入口 | 门面、简化调用 |
| 享元 | 共享细粒度对象 | 共享、节省内存 |
| 代理 | 控制访问 | 权限控制、远程代理、虚代理 |

#### 4.3 行为型模式

| 模式 | 用途 | 关键词 |
|---|---|---|
| 观察者 | 一对多通知 | 订阅、监听、通知 |
| 策略 | 动态切换算法 | 支付方式、算法切换 |
| 状态 | 状态变化导致行为变化 | 状态切换 |
| 责任链 | 请求沿链传递 | 审批流、逐级处理 |
| 命令 | 把请求封装成对象 | 撤销、排队、日志 |
| 模板方法 | 固定流程，细节延迟 | 算法骨架 |

> 速记：**创建单工建原，结构适桥组装外享代，行为观察策状责命模。**

### 5. 架构评估方法

| 方法 | 全称 | 主要目标 | 关键词 |
|---|---|---|---|
| ATAM | Architecture Tradeoff Analysis Method | 评估质量属性之间的折衷 | 折中、权衡、风险点、敏感点 |
| SAAM | Software Architecture Analysis Method | 分析架构可修改性 | 场景、可修改性 |
| CBAM | Cost Benefit Analysis Method | 从成本收益角度选择架构策略 | 成本、收益、经济性 |

### 6. DSSA（领域特定软件架构）

DSSA 专门针对某一类应用领域建立可复用架构。

#### 三个基本活动

1. **领域分析**：识别领域共同需求，建立领域模型
2. **领域设计**：形成该领域的软件架构
3. **领域实现**：开发可复用构件并组织复用资产

#### 参与角色

- 领域专家：懂业务
- 领域分析师：提炼需求
- 架构设计人员：设计架构
- 实现人员：开发可复用组件

## 三、软件工程与项目管理（最稳拿分区）

### 1. CMMI 五级成熟度模型

> 口诀：**初可已量优**。

| 级别 | 名称 | 特征 |
|---|---|---|
| 1 | 初始级 | 过程无序，依赖个人能力 |
| 2 | 可重复级 | 有项目管理，能计划、跟踪 |
| 3 | 已定义级 | 形成组织级标准过程 |
| 4 | 量化管理级 | 用数据度量和控制过程 |
| 5 | 优化级 | 持续改进，主动缺陷预防 |

### 2. 常见开发模型对比

| 模型 | 特征 | 适用场景 | 易错点 |
|---|---|---|---|
| 瀑布模型 | 线性推进，上阶段完成再下阶段 | 需求稳定、变更少 | 反馈晚，风险后置 |
| 原型模型 | 先做原型帮助澄清需求 | 需求不清晰 | 原型不等于最终产品 |
| 增量模型 | 分批交付功能 | 可分阶段上线 | 每个增量都应可运行 |
| 螺旋模型 | 风险驱动，迭代开发 | 大型高风险项目 | 核心关键词是风险分析 |
| 敏捷模型 | 快速迭代，持续反馈 | 需求变化快 | 不是不要文档，而是轻文档重协作 |

### 3. 软件测试分类

| 测试类型 | 主要对象 | 主要发现的错误 |
|---|---|---|
| 单元测试 | 模块/函数 | 编码错误、详细设计错误 |
| 集成测试 | 模块接口 | 接口错误、概要设计错误 |
| 确认测试 | 需求满足程度 | 是否符合需求规格说明书 |
| 系统测试 | 完整系统 | 性能、安全、兼容、恢复等问题 |
| 验收测试 | 用户/甲方角度 | 是否满足交付要求 |

### 4. 项目管理核心工具

| 概念 | 定义 | 高频考点 |
|---|---|---|
| WBS | 按交付成果分解工作 | 不能按部门分解 |
| 工作包 | WBS 最底层可管理单元 | 可估算、可分配、可跟踪 |
| 关键路径 | 决定工期的最长路径 | 总时差为 0 |
| 甘特图 | 条形进度图 | 适合看进度，不擅长复杂依赖 |
| PERT 图 | 网络计划图 | 适合不确定时间估算 |

#### PERT 三点估算

如果题目给：

- 乐观时间 `to`
- 最可能时间 `tm`
- 悲观时间 `tp`

则期望工期为：

`te = (to + 4tm + tp) / 6`

## 四、数据库知识（高频必考）

### 1. 事务 ACID

> 口诀：**原一隔持**。

| 特性 | 含义 | 常考保障手段 |
|---|---|---|
| 原子性 Atomicity | 要么全做，要么全不做 | 回滚、撤销日志 |
| 一致性 Consistency | 事务前后数据必须保持正确 | 完整性约束、正确程序逻辑 |
| 隔离性 Isolation | 并发事务互不干扰 | 锁、MVCC、并发控制 |
| 持久性 Durability | 提交后永久生效 | 重做日志、落盘 |

### 2. 并发问题与隔离级别

#### 常见并发问题

- **脏读**：读到了别人未提交的数据
- **不可重复读**：同一事务两次读同一行，结果不同
- **幻读**：同一事务两次按条件查询，返回的记录数不同

#### 隔离级别（从低到高）

1. 读未提交
2. 读已提交
3. 可重复读
4. 串行化

> 隔离越强，并发性能通常越低。

### 3. 数据库范式

> 口诀：**1NF 列不可分，2NF 消部分，3NF 消传递，BCNF 更严格。**

| 范式 | 要点 | 记忆方法 |
|---|---|---|
| 1NF | 字段不可再分 | 每列保持原子性 |
| 2NF | 消除对复合主键的部分依赖 | 非主属性必须完全依赖主键 |
| 3NF | 消除传递依赖 | 非主属性不能依赖非主属性 |
| BCNF | 每个决定因素都必须是候选键 | 比 3NF 更严格 |

### 4. 关系运算与索引

#### 常见关系运算

- 选择：从行角度筛选
- 投影：从列角度选择
- 连接：把多个表按条件拼起来

#### 索引常见考点

- 索引可以提升查询性能，但会降低插入/更新性能
- 适合为高频查询条件建立索引
- 不是索引越多越好

## 五、操作系统知识（必考基础）

### 1. 进程、线程、程序

| 概念 | 含义 |
|---|---|
| 程序 | 静态代码和指令集合 |
| 进程 | 程序的一次执行过程，是资源分配单位 |
| 线程 | CPU 调度单位，是进程中的执行流 |

### 2. 死锁四大必要条件

> 口诀：**互斥、请求保持、不剥夺、环路等待。**

1. 互斥条件
2. 请求和保持条件
3. 不剥夺条件
4. 环路等待条件

#### 记忆结论

- 四个条件**缺一不可**
- 打破任意一个条件，都能避免死锁
- **银行家算法**属于死锁避免算法

### 3. 存储管理常考点

- **页式存储**：离散分配，解决连续内存碎片问题
- **段式存储**：按逻辑模块划分，更符合程序逻辑
- **虚拟存储**：逻辑上扩大内存容量，本质依赖局部性原理

### 4. 常见调度指标

- 周转时间
- 带权周转时间
- 响应时间
- 吞吐量
- CPU 利用率

## 七、安全与标准化（适合临考冲刺）

### 1. 网络安全等级保护 2.0

| 等级 | 名称 | 常见场景 |
|---|---|---|
| 一级 | 自主保护 | 小型一般系统 |
| 二级 | 指导保护 | 一般企业信息系统 |
| 三级 | 监督保护 | 金融、电信、能源等重点行业 |
| 四级 | 强制保护 | 重要国家基础设施 |
| 五级 | 专控保护 | 国防、核心机密领域 |

#### 三级系统常考关键词

- 自主访问控制
- 强制访问控制
- 安全审计
- 入侵防范
- 数据完整性保护

> 实战中常考：**三级是重点，记住“金融、电信、能源”**。

### 2. 密码学与安全协议

#### 2.1 对称加密

- 常见：DES、3DES、AES、SM4
- 特点：**速度快**
- 缺点：密钥分发困难

#### 2.2 非对称加密

- 常见：RSA、ECC、SM2
- 特点：公钥和私钥成对出现
- 用途：密钥交换、数字签名、身份认证

#### 2.3 摘要算法

- 常见：MD5、SHA 系列、SM3
- 特点：不可逆
- 用途：完整性校验

#### 2.4 数字签名

> 口诀：**私钥签名，公钥验证。**

作用：

- 验证身份
- 防止篡改
- 抗抵赖

#### 2.5 加密与签名的区别

- **保密**：公钥加密，私钥解密
- **认证/防伪**：私钥签名，公钥验证

#### 2.6 SSL/TLS

- 工作在传输层之上、应用层之下
- 用于建立安全通信通道
- HTTPS 就是 HTTP + TLS/SSL

## 八、计算机组成与网络基础

### 1. 计算机组成原理

- **冯·诺依曼结构**：程序和数据同存储器，按指令顺序执行
- **控制器**：负责指令控制
- **运算器**：负责算术和逻辑运算
- **存储器**：保存程序和数据
- **输入/输出设备**：与外界交互

#### 常见寄存器

| 寄存器 | 作用 |
|---|---|
| PC | 保存下一条将执行指令的地址 |
| IR | 保存当前指令 |
| ACC | 累加器，保存运算结果 |

#### Cache

- 位于 CPU 与主存之间
- 作用：提高访问速度
- 基础原理：程序访问具有局部性

### 2. OSI 与 TCP/IP 常考协议

> 记忆主线：**网层看 IP，传层看 TCP/UDP，应层看 HTTP/DNS/FTP/SMTP。**

| 层次 | 常见协议 | 核心功能 |
|---|---|---|
| 数据链路层 | ARP | IP 地址解析为 MAC 地址 |
| 网络层 | IP、ICMP、IGMP | 路由转发、差错控制 |
| 传输层 | TCP、UDP | 端到端传输 |
| 应用层 | HTTP、HTTPS、FTP、SMTP、POP3、DNS | 面向具体应用 |

### 3. 重点协议辨析

| 协议 | 层次 | 特点 |
|---|---|---|
| TCP | 传输层 | 可靠、面向连接、有流量控制和拥塞控制 |
| UDP | 传输层 | 不可靠、无连接、快 |
| HTTP | 应用层 | 超文本传输，默认端口 80 |
| HTTPS | 应用层 | 安全超文本传输，默认端口 443 |
| DNS | 应用层 | 域名解析，常用 UDP 53 |
| ARP | 数据链路层/网络接口层常见考法 | 局域网内 IP 转 MAC |

#### 高频易错点

- ARP 只能在**局域网**中使用
- DNS 常用 UDP，不代表绝对不用 TCP
- TCP 可靠不代表更快

## 九、系统可靠性与性能计算（必背公式）

### 1. 可靠性基本概念

- **可靠性**：系统在规定条件和时间内完成规定功能的能力
- **可用性**：系统可正常使用的概率/时间比例

### 2. 串联系统与并联系统

> 口诀：**串联乘，并联减。**

#### 串联系统

只要一个部件坏了，系统就坏：

`R = R1 × R2 × ... × Rn`

#### 并联系统

只有所有部件都坏了，系统才坏：

`R = 1 - (1-R1)(1-R2)...(1-Rn)`

#### 结论

- 串联越多，可靠性越低
- 并联冗余越多，可靠性越高

### 3. 平均无故障时间

- **MTBF**：平均无故障时间
- 数值越大，说明系统越稳定

### 4. 可用性公式

常考形式：

`可用性 = MTBF / (MTBF + MTTR)`

其中：

- MTBF：平均无故障时间
- MTTR：平均修复时间

## 十、知识产权与法律法规（高频常识题）

### 1. 保护期限

| 类型 | 期限 | 备注 |
|---|---|---|
| 发明专利权 | 20 年 | 从申请日起算 |
| 实用新型专利权 | 10 年 | 从申请日起算 |
| 外观设计专利权 | 15 年 | 从申请日起算 |
| 商标权 | 10 年 | 可续展 |
| 软件著作权 | 作者终生及死后 50 年 | 单位作品一般为首次发表后 50 年 |

### 2. 著作权归属高频考点

#### 职务开发

- 一般利用单位资源、按工作职责完成的软件
- **著作权归单位**
- 开发者通常享有署名权

#### 委托开发

- **有约定按约定**
- **无约定通常归受托人**

### 3. 常见陷阱

- **商标权**可以不断续展，不是一次到期永久失效
- **软件著作权**保护的是表达，不保护思想本身
- **专利权**强调技术方案的法律保护

## 十二、综合知识高频易错点汇总

### 1. 概念类易错点

- 微服务 ≠ 分布式
- 敏捷 ≠ 不写文档
- 负载均衡 ≠ 只提升性能
- 消息队列 ≠ 事件驱动架构本身
- 数字签名 ≠ 加密本身

### 2. 项目管理类易错点

- WBS 一定是**按交付成果分解**
- 关键路径上的总时差一定为 0
- 甘特图擅长进度展示，不擅长复杂逻辑依赖表达

### 3. 安全类易错点

- 数字签名保障的是**完整性、认证性、抗抵赖**
- 摘要算法用于完整性校验，不负责保密
- SSL/TLS 主要解决传输过程中的安全通信问题

### 4. 网络类易错点

- ARP 不能跨路由器传播
- DNS 虽常用 UDP，但大报文/区域传送也可用 TCP
- UDP 不可靠，但实时性高
