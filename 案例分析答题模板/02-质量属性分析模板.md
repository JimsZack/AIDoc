# 质量属性分析 — 案例答题模板

> 高频考法：给出系统描述和需求场景，要求识别质量属性、分析质量属性场景、给出满足质量属性的架构策略/战术。

---

## 模板一：识别质量属性

### 题目类型

> "根据题目描述，识别该系统涉及的质量属性，并说明依据。"

### 答题框架

```
【答】
根据题目描述，该系统涉及以下质量属性：

（1）性能（Performance）
  依据：题目中提到"[引用题干原文，如：系统需支持10000并发用户]"，这属于性能中的[吞吐量/响应时间/并发数]需求。

（2）可用性（Availability）
  依据：题目中提到"[引用题干原文，如：系统需7×24小时不间断运行]"，这属于可用性需求。

（3）安全性（Security）
  依据：题目中提到"[引用题干原文，如：用户数据需加密传输]"，这属于安全性中的[机密性/完整性/不可否认性]需求。

（4）可修改性（Modifiability）
  依据：题目中提到"[引用题干原文，如：系统需要能快速接入新的支付渠道]"，这属于可修改性需求。

...（根据题目实际情况列举）
```

### 质量属性识别速查表

| 题干关键词 | 对应质量属性 |
|------------|-------------|
| 响应时间、并发数、吞吐量、延迟 | 性能 |
| 不间断运行、故障恢复、冗余备份、自动切换 | 可用性 |
| 加密、认证、授权、防攻击、审计 | 安全性 |
| 易扩展、新增功能、接入新模块、配置化 | 可修改性 |
| 可观测、日志、Mock测试、接口测试 | 可测试性 |
| 水平扩展、弹性伸缩、按需扩容 | 可伸缩性 |
| 第三方对接、协议转换、异构互通 | 互操作性 |
| 易学习、用户体验、操作简便 | 易用性 |

## 模板二：质量属性场景描述（六要素）

### 题目类型

> "请用质量属性场景描述该系统的[某]需求。"

### 答题框架

```
【答】
该系统的 [XX质量属性] 场景描述如下：

| 场景要素 | 具体描述 |
|----------|----------|
| 刺激源（Source） | [产生刺激的实体，如：外部用户/攻击者/硬件故障] |
| 刺激（Stimulus） | [发生了什么事，如：发起大量并发请求/服务器宕机/恶意SQL注入] |
| 环境（Environment） | [系统当前状态，如：正常运行/过载/降级模式] |
| 制品（Artifact） | [被影响的对象，如：Web服务器/数据库/某模块] |
| 响应（Response） | [系统如何反应，如：在2秒内返回结果/自动切换到备机] |
| 响应度量（Measure） | [可量化的指标，如：99.9%请求在2s内响应/故障恢复时间<30s] |
```

### 实战示例

**场景**：电商系统在双十一大促期间的性能需求

```
【答】
该系统的性能质量属性场景描述如下：

| 场景要素 | 具体描述 |
|----------|----------|
| 刺激源 | 大量在线用户（双十一活动参与者） |
| 刺激 | 同时发起商品查询和下单请求，并发量达到平时的10倍 |
| 环境 | 系统处于峰值负载状态 |
| 制品 | 订单服务、商品查询服务 |
| 响应 | 系统正常处理请求，核心交易链路不超时 |
| 响应度量 | 商品查询响应时间≤500ms，下单成功率≥99.5% |
```

## 模板三：给出架构策略/战术

### 题目类型

> "针对该系统的[某质量属性]需求，给出满足该需求的架构策略。"

### 答题框架

```
【答】
针对该系统的 [XX质量属性] 需求，建议采用以下架构策略：

（1）[策略名称]
  具体措施：[怎么做]
  预期效果：[解决什么问题]

（2）[策略名称]
  具体措施：[怎么做]
  预期效果：[解决什么问题]

（3）[策略名称]
  具体措施：[怎么做]
  预期效果：[解决什么问题]
```

### 各质量属性的标准架构策略/战术（答题素材库）

#### 性能策略

| 策略 | 具体做法 | 适用场景 |
|------|----------|----------|
| 资源调度 | 引入负载均衡，将请求分发到多个服务实例 | 高并发 |
| 缓存 | 对热点数据使用 Redis/Memcached 缓存 | 频繁读取 |
| 异步处理 | 使用消息队列将耗时操作异步化 | 削峰填谷 |
| 数据库优化 | 读写分离、分库分表、索引优化 | 数据量大 |
| CDN加速 | 静态资源部署到CDN节点 | 全国/全球访问 |
| 连接池 | 数据库/HTTP连接池复用 | 频繁连接 |

#### 可用性策略

| 策略 | 具体做法 | 适用场景 |
|------|----------|----------|
| 冗余 | 主备/主从/集群部署，消除单点故障 | 7×24小时 |
| 心跳检测 | 定期检测服务存活状态 | 快速发现故障 |
| 自动故障切换 | 主节点故障后自动切换到备节点 | 高可用 |
| 服务降级 | 非核心功能降级，保证核心链路 | 过载保护 |
| 熔断机制 | 依赖服务异常时快速熔断，防止级联故障 | 微服务 |
| 限流 | 控制请求速率，防止系统被压垮 | 突发流量 |

#### 安全性策略

| 策略 | 具体做法 | 适用场景 |
|------|----------|----------|
| 身份认证 | 采用 OAuth2.0/JWT 等认证机制 | 用户登录 |
| 访问控制 | 基于角色的权限控制(RBAC) | 权限管理 |
| 数据加密 | TLS传输加密 + AES数据加密 | 敏感数据 |
| 输入验证 | 对用户输入进行合法性校验和过滤 | 防注入攻击 |
| 审计日志 | 记录关键操作日志，支持事后追溯 | 合规审计 |
| 安全网关 | 统一入口进行安全检查和流量清洗 | 边界防护 |

#### 可修改性策略

| 策略 | 具体做法 | 适用场景 |
|------|----------|----------|
| 模块化 | 高内聚低耦合的模块划分 | 减少变更影响 |
| 接口隔离 | 通过接口/抽象层隔离变化 | 实现可替换 |
| 配置化 | 将易变参数外置为配置文件 | 无需改代码 |
| 插件机制 | 支持动态加载新功能模块 | 功能扩展 |
| 依赖反转 | 高层模块不依赖低层模块的具体实现 | 解耦 |
| 中间件/适配层 | 通过适配层隔离外部系统变化 | 第三方对接 |

## 模板四：质量属性权衡分析

### 题目类型

> "该系统同时要求高性能和高安全性，请分析两者之间的冲突及权衡方案。"

### 答题框架

```
【答】
（1）冲突分析：
  [质量属性A] 和 [质量属性B] 之间存在冲突：
  - 为满足A需要 [某措施]，但该措施会导致B [具体负面影响]
  - 具体表现为：[举例说明冲突场景]

（2）权衡方案：
  ① [方案描述]：在 [条件/场景] 下优先保证A，在 [条件/场景] 下优先保证B
  ② [方案描述]：采用 [折中技术手段]，兼顾两者
  ③ [方案描述]：通过 [分层/分区策略]，对不同部分采用不同策略

（3）结论：
  综合考虑本系统的业务特点（[引用题干]），建议以 [某质量属性] 为首要保障，
  通过 [具体措施] 在可接受范围内兼顾 [另一质量属性]。
```

### 常见质量属性冲突

| 冲突对 | 冲突原因 | 常见权衡手段 |
|--------|----------|-------------|
| 性能 vs 安全性 | 加密/认证增加处理开销 | 分级加密、硬件加速 |
| 性能 vs 可修改性 | 解耦增加间接调用层次 | 热路径直连、冷路径解耦 |
| 可用性 vs 一致性 | CAP定理约束 | 最终一致性、分区策略 |
| 安全性 vs 易用性 | 安全措施增加操作步骤 | 风险自适应认证 |
| 可修改性 vs 性能 | 抽象层增加运行开销 | 关键路径优化、AOP |

---

## 答题注意事项

1. **识别题必须引用题干原文**：不能凭空说"该系统需要高性能"，要指出题目哪句话体现了性能需求
2. **策略题要具体**：不能只说"使用缓存"，要说"对用户频繁查询的商品信息使用Redis缓存"
3. **场景六要素题务必写全**：漏一个要素就少一个得分点
4. **权衡题要有结论**：不能只分析冲突不给解决方案
5. **注意题目给的约束条件**：如预算、团队规模、时间要求等都可能影响策略选择
