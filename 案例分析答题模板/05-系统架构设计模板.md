# 系统架构设计 — 案例答题模板

> 高频考法：给出系统需求描述，要求进行架构设计（模块划分、技术选型、部署方案、数据流设计等），或对已有设计进行分析。

---

## 模板一：系统架构整体设计

### 题目类型

> "请为该系统设计一个合理的系统架构，并说明设计理由。"

### 答题框架

```
【答】
（1）总体架构方案：
  推荐采用 [架构风格] 架构，整体分为以下层次/服务：
  - [层次/模块1]：负责 [职责描述]
  - [层次/模块2]：负责 [职责描述]
  - [层次/模块3]：负责 [职责描述]
  - ...

（2）架构设计说明：

  ① 表现层/接入层：
    - 采用 [技术方案]，负责 [具体功能]
    - 设计理由：[为什么这样设计]

  ② 业务逻辑层/服务层：
    - 划分为 [服务/模块列表]
    - 服务间通过 [通信方式] 进行交互
    - 设计理由：[为什么这样划分]

  ③ 数据层：
    - 采用 [数据库/存储方案]
    - 数据访问通过 [方式] 实现
    - 设计理由：[为什么选择该方案]

  ④ 基础设施层（如涉及）：
    - 包括 [中间件/框架/工具列表]
    - 设计理由：[为什么需要]

（3）关键设计决策：
  ① [决策1]：选择 [方案A] 而非 [方案B]，因为 [理由]
  ② [决策2]：...
  ③ [决策3]：...
```

## 模板二：微服务架构设计

### 题目类型

> "请对该系统进行微服务拆分，并设计服务间的通信机制。"

### 答题框架

```
【答】
（1）服务拆分方案：
  根据业务领域划分，将系统拆分为以下微服务：

| 服务名称 | 职责 | 核心功能 | 数据存储 |
|----------|------|----------|----------|
| [服务A] | ... | ①... ②... | [独立数据库] |
| [服务B] | ... | ①... ②... | [独立数据库] |
| [服务C] | ... | ①... ②... | [独立数据库] |

  拆分依据：按照 [业务领域/DDD限界上下文/数据边界] 进行拆分。

（2）服务间通信设计：
  ① 同步通信：[服务A] 调用 [服务B] 采用 RESTful API/gRPC，用于 [场景]
  ② 异步通信：[服务C] 通过消息队列(如 Kafka/RabbitMQ)发布事件，[服务D] 订阅处理，用于 [场景]

（3）服务治理设计：
  ① API网关：统一入口，负责路由、限流、认证
  ② 服务注册与发现：使用 [Nacos/Eureka/Consul]，实现服务动态注册
  ③ 熔断降级：使用 [Sentinel/Hystrix]，防止级联故障
  ④ 配置中心：使用 [Nacos/Apollo]，统一配置管理
  ⑤ 链路追踪：使用 [Zipkin/SkyWalking]，实现全链路可观测

（4）数据一致性方案：
  - 强一致场景：采用 [分布式事务/Seata/TCC]
  - 最终一致场景：采用 [事件驱动/Saga模式/本地消息表]
```

  │           用户/客户端                 │

  │     [服务器数量/容器实例数]          │
  └──────────────┬──────────────────────┘
                 ↓
  ┌─────────────────────────────────────┐

  ① [缓存层设计]

（4）监控运维：

## 常见架构设计题型速查

### Web 系统典型分层

```
用户浏览器/移动端
    ↓
CDN + 负载均衡（Nginx）
    ↓
API 网关（统一认证、限流、路由）
    ↓
微服务集群（业务服务A、服务B、服务C...）
    ↓
中间件层（Redis缓存、MQ消息队列、ES搜索）
    ↓
数据层（MySQL主从、MongoDB、对象存储）
    ↓
基础设施（K8s容器编排、监控、日志）
```

### 大数据系统典型架构

```
数据源（业务数据库、日志、IoT设备）
    ↓
数据采集（Flume/Canal/Kafka Connect）
    ↓
消息中间件（Kafka）
    ↓  ↓
实时计算（Flink/Spark Streaming） | 离线计算（Spark/Hive）
    ↓                                ↓
数据存储（HBase/HDFS/ClickHouse）
    ↓
数据服务（API/报表/大屏）
```

## 答题注意事项

1. **架构图尽量画出来**：哪怕是简单的框图+箭头，比纯文字更清晰
2. **分层/分模块要有依据**：说明为什么这样拆分，对应什么设计原则
3. **技术选型不能空泛**：要针对题目场景的具体需求来选
4. **注意非功能需求**：性能、安全、可用性等方面的设计不能遗漏
5. **考虑演进性**：好的架构答案会体现对未来扩展的考虑
6. **符合题目约束**：如果题目提到预算有限/团队小，方案要匹配
