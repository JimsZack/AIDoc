# 架构评估（ATAM/SAAM）— 案例答题模板

> 高频考法：给出系统架构描述，要求运用 ATAM 或 SAAM 方法进行评估，识别风险点、敏感点、权衡点等。

---

## 模板一：ATAM 评估过程

### 题目类型

> "请使用 ATAM 方法对该系统架构进行评估。"  
> "请识别该架构中的风险点、敏感点和权衡点。"

### ATAM 核心概念速记

| 概念 | 定义 | 答题关键 |
|------|------|----------|
| **敏感点** | 一个架构决策影响了某个质量属性的实现 | "如果改变X，会影响Y" |
| **权衡点** | 一个架构决策同时影响多个质量属性（有利有弊） | "X有利于A但不利于B" |
| **风险点** | 可能导致质量属性无法满足的架构决策 | "X可能导致Y不达标" |
| **非风险点** | 确认可行的架构决策 | "X可以保证Y" |

### 答题框架

```
【答】
使用 ATAM 方法对该系统架构进行评估，分析如下：

一、敏感点（Sensitivity Points）：
（1）[架构决策描述]是[质量属性]的敏感点。
    分析：如果[改变/调整该决策]，将直接影响系统的[质量属性表现]。
    例如：[结合题干具体说明]

（2）...

二、权衡点（Tradeoff Points）：
（1）[架构决策描述]是性能和[另一质量属性]之间的权衡点。
    分析：该决策有利于提升[质量属性A]，但同时会降低[质量属性B]。
    具体表现：[结合题干说明]

（2）...

三、风险点（Risk Points）：
（1）[架构决策描述]存在风险。
    风险描述：在[某种条件/场景]下，该决策可能导致[质量属性]无法满足需求。
    建议：[改进建议]

（2）...

四、非风险点（Non-risk Points）：
（1）[架构决策描述]是非风险点。
    分析：该决策能够有效保证[质量属性]的实现，因为[原因]。
```

### 实战示例

**题目场景**：某在线支付系统采用三层架构，数据库使用主从复制，Web层使用Nginx负载均衡，业务层采用同步调用。

```
【答】
使用 ATAM 方法对该系统架构进行评估：

一、敏感点：
（1）数据库主从复制的同步延迟是可用性的敏感点。
    分析：主从同步延迟的大小直接影响故障切换后数据的完整性，延迟越大，切换后丢失数据越多。

（2）Nginx的负载均衡策略是性能的敏感点。
    分析：不同的负载均衡算法（轮询/权重/最少连接）直接影响请求分配的均匀性和响应时间。

二、权衡点：
（1）业务层的同步调用方式是性能与数据一致性之间的权衡点。
    分析：同步调用保证了数据强一致性，但在高并发场景下会增加响应时间，降低系统吞吐量。

三、风险点：
（1）业务层采用同步调用在高并发场景下存在性能风险。
    风险描述：当并发量超过系统容量时，同步调用可能导致线程池耗尽，系统响应超时。
    建议：对非核心业务引入异步消息队列处理。

四、非风险点：
（1）使用Nginx做负载均衡是非风险点。
    分析：Nginx是成熟的高性能负载均衡方案，可以有效分发请求，保证系统性能。
```

## 模板二：SAAM 评估过程

### 题目类型

> "请使用 SAAM 方法评估该架构的可修改性。"

### SAAM 评估步骤

| 步骤 | 内容 | 答题要点 |
|------|------|----------|
| 1 | 描述候选架构 | 简述架构的关键组件和连接件 |
| 2 | 开发场景 | 列出变更场景（功能变更/性能提升等） |
| 3 | 评估单个场景 | 分析每个场景是直接支持还是需要修改 |
| 4 | 场景交互 | 分析多个场景是否影响同一组件 |
| 5 | 总体评估 | 给出架构可修改性的综合评价 |

### 答题框架

```
【答】
使用 SAAM 方法评估该架构，过程如下：

（1）架构描述：
  该系统由 [组件A]、[组件B]、[组件C] 等核心组件组成，
  组件间通过 [连接方式] 进行通信。

（2）变更场景列举：
  场景1：[描述一个可能的变更需求]
  场景2：[描述另一个可能的变更需求]
  场景3：...

（3）场景评估：

| 场景 | 是否直接支持 | 需要修改的组件 | 修改难度 |
|------|-------------|---------------|----------|
| 场景1 | 是/否 | [组件名] | 低/中/高 |
| 场景2 | 是/否 | [组件名] | 低/中/高 |
| 场景3 | 是/否 | [组件名] | 低/中/高 |

（4）场景交互分析：
  - 场景X和场景Y都需要修改[组件Z]，存在场景交互
  - 该组件成为变更的"热点"，需要重点关注其可修改性设计

（5）综合评估：
  该架构对[某类变更]具有良好的可修改性，因为[原因]；
  但对[另一类变更]的支持较弱，建议[改进措施]。
```

## ATAM 九步流程（背诵要点）

如果题目考 ATAM 的完整流程：

```
第1步：介绍 ATAM 方法（评估团队向参与者说明方法）
第2步：介绍业务驱动因素（项目决策者描述业务目标）
第3步：介绍架构（架构师描述架构）
第4步：识别架构方法（识别架构中使用的架构模式/风格）
第5步：生成质量属性效用树（将质量属性分解为具体场景并排优先级）
第6步：分析架构方法（通过场景分析识别风险点、敏感点、权衡点）
第7步：头脑风暴确定场景（更多利益相关者参与产生新场景）
第8步：分析架构方法（对新场景重复第6步分析）
第9步：呈现结果（整理所有风险点、敏感点、权衡点并报告）
```

### 效用树模板

```
                         系统质量
                       /     |      \
                  性能      可用性    安全性
                 /    \      |        |
           响应时间  吞吐量  故障恢复  数据加密
              |       |       |        |
         (H,H)    (H,M)   (H,H)    (M,H)
     "首页加载  "支持1万  "故障后30s "所有传输
      <2s"     并发"     内恢复"    TLS加密"

注：(重要性, 实现难度)  H=高, M=中, L=低
```

## 答题注意事项

1. **区分敏感点和权衡点**：敏感点影响一个属性，权衡点影响多个属性（有正有负）
2. **风险点要可验证**：描述具体在什么条件下会出问题，不要泛泛而谈
3. **ATAM 和 SAAM 不要混淆**：ATAM 侧重质量属性权衡，SAAM 侧重可修改性
4. **效用树题一定要写优先级**：用(H,M,L)标注重要性和实现难度
5. **结合题干具体架构决策**：不要脱离题目给的具体技术方案空谈
