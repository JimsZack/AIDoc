# 设计模式应用 — 案例答题模板

> 高频考法：给出系统设计需求或类图，要求识别适用的设计模式、说明选用理由、补全类图/画出结构图。

---

## 模板一：选择设计模式并说明理由

### 题目类型

> "根据以下需求，请选择合适的设计模式并说明理由。"

### 答题框架

```
【答】
（1）推荐使用 [模式名称]（[所属类别：创建型/结构型/行为型]）。

（2）模式意图：
  [模式名称] 的核心意图是 [一句话描述模式目的]。

（3）选择理由：
  ① 题目需求中提到"[引用题干]"，这与该模式的 [核心特征] 匹配。
  ② 该模式通过 [机制描述] 解决了 [具体问题]。
  ③ 使用该模式可以带来 [好处：如解耦/复用/扩展性]。

（4）模式结构说明：
  - [角色A]：负责 [职责]，对应题目中的 [具体组件/类]
  - [角色B]：负责 [职责]，对应题目中的 [具体组件/类]
  - [角色C]：负责 [职责]，对应题目中的 [具体组件/类]

（5）不选其他模式的原因（可选）：
  - 不选 [模式X]：因为 [原因]
```


### 答题技巧

- **先识别模式**：从类图结构和关系判断是哪种设计模式
- **再对号入座**：把类图中的类/接口对应到模式的标准角色
- **关注箭头方向**：实线=关联/组合，虚线=依赖/实现，空心三角=继承

## 高频设计模式案例答题素材

### 创建型模式

#### 工厂方法 vs 抽象工厂

| 维度 | 工厂方法 | 抽象工厂 |
|------|----------|----------|
| 创建对象 | 一种产品 | 一族产品 |
| 扩展方式 | 新增具体工厂子类 | 新增工厂实现类 |
| 题干关键词 | "统一创建接口""延迟到子类" | "产品族""一组相关对象" |
| 典型场景 | 日志记录器工厂 | 跨平台UI组件工厂 |

#### 单例模式

```
适用题干：
- "全局只有一个实例"
- "共享资源的统一管理"
- "配置管理器""连接池""线程池"

答题要点：
- 私有构造函数
- 静态实例引用
- 公共访问方法
- 线程安全（双重检查锁定/静态内部类）
```

### 结构型模式

#### 适配器 vs 装饰器 vs 代理

| 维度 | 适配器 | 装饰器 | 代理 |
|------|--------|--------|------|
| 目的 | 接口转换 | 增强功能 | 控制访问 |
| 关键词 | "旧系统对接""接口不兼容" | "动态添加功能""不修改原类" | "权限控制""远程访问""延迟加载" |
| 结构特征 | 包装者和被包装者接口不同 | 包装者和被包装者接口相同 | 代理和真实对象接口相同 |

#### 外观模式

```
适用题干：
- "简化子系统调用"
- "统一接口/入口"
- "降低客户端与子系统的耦合"

答题结构：
- Facade（外观类）：提供统一的高层接口
- SubSystemA/B/C：各子系统类
- Client：只与Facade交互
```

#### 桥接模式

```
适用题干：
- "多维度变化"
- "抽象部分与实现部分独立变化"
- 出现类似 "不同形状 × 不同颜色" 的组合

答题结构：
- Abstraction（抽象类）
- RefinedAbstraction（扩展抽象）
- Implementor（实现接口）
- ConcreteImplementor（具体实现）
```

### 行为型模式

#### 观察者模式

```
适用题干：
- "一对多通知"
- "状态变化时通知所有相关对象"
- "发布/订阅"
- "事件监听"

答题结构：
- Subject（被观察者）：维护观察者列表，通知变更
- Observer（观察者接口）：定义更新方法
- ConcreteSubject：具体被观察者
- ConcreteObserver：具体观察者
```

#### 策略模式

```
适用题干：
- "多种算法/策略可切换"
- "运行时选择不同实现"
- "支付方式选择""排序算法切换"

答题结构：
- Context（上下文）：持有策略引用
- Strategy（策略接口）：定义算法接口
- ConcreteStrategyA/B/C：具体策略
```

#### 状态模式 vs 策略模式

| 维度 | 状态模式 | 策略模式 |
|------|----------|----------|
| 驱动因素 | 对象内部状态变化 | 外部主动选择 |
| 切换方式 | 自动切换（状态内部决定） | 客户端指定 |
| 关键词 | "状态流转""不同状态不同行为" | "选择算法""切换方式" |

#### 责任链模式

```
适用题干：
- "逐级审批"
- "多个处理者依次处理"
- "请求沿链传递直到被处理"

答题结构：
- Handler（处理者接口）：定义处理方法和后继引用
- ConcreteHandlerA/B/C：具体处理者
- Client：向链头提交请求
```

#### 模板方法模式

```
适用题干：
- "算法骨架固定，某些步骤由子类实现"
- "流程固定，细节可变"
- "钩子方法"

答题结构：
- AbstractClass：定义模板方法（final）和抽象步骤
- ConcreteClassA/B：实现抽象步骤
```

## 设计模式七大原则（答题加分）

回答"为什么选择该模式"时，可以引用设计原则：

| 原则 | 核心含义 | 答题用法 |
|------|----------|----------|
| 开闭原则 | 对扩展开放，对修改关闭 | "使用XX模式后，新增功能只需添加新类，无需修改现有代码" |
| 单一职责 | 一个类只有一个变化原因 | "通过XX模式将职责分离到不同类中" |
| 里氏替换 | 子类可替换父类 | "具体实现类可以互相替换，不影响客户端" |
| 依赖倒转 | 依赖抽象不依赖具体 | "客户端依赖接口而非具体实现，便于替换" |
| 接口隔离 | 接口要小而专 | "将大接口拆分为多个专用接口" |
| 迪米特法则 | 最少知识原则 | "通过外观类减少了客户端需要了解的类数量" |
| 合成复用 | 优先组合而非继承 | "使用对象组合替代类继承，更灵活" |

## 答题注意事项

1. **先判断模式类别再具体定位**：创建型→看对象创建方式，结构型→看类组合关系，行为型→看对象交互
2. **模式名称要准确**：不能写错名字或混淆中英文
3. **类图填空题注意接口和类的区别**：接口用虚线实现，类用实线继承
4. **说明模式好处时结合题目**：不要只背定义
5. **如果要画类图**：注意画清楚类名、方法名、关系箭头
