# 质量属性与架构策略案例 — 练习题

---

## 案例一：电商平台质量属性分析

### 题目背景

某大型电商平台即将迎来"双十一"促销活动。根据以往数据，活动期间系统并发量是平时的 50 倍。架构师小王正在对系统进行压测和架构评审，发现了以下需求和问题：

**系统当前状况：**
- 现有架构：微服务架构，服务间同步调用
- 数据库：MySQL 主从集群
- 缓存：部分热点数据使用 Redis
- 部署：云服务器，手动运维

**已知问题：**
1. 压测发现订单服务在 10 万并发时响应时间超过 5 秒（目标要求 2 秒以内）
2. 历史记录显示，大促期间支付网关曾出现故障，导致整个购物流程不可用
3. 发现 SQL 注入攻击日志，数据库存在安全风险
4. 运营人员反映新功能上线后经常出现回归问题，测试覆盖率不足

---

### 问题一（10分）

请识别上述系统中涉及的质量属性，并为每种质量属性给出至少一条架构改进策略（结合题目背景说明）。

**参考答案：**

```
（1）性能（Performance）
  依据：压测发现订单服务 10 万并发时响应时间超过 5 秒，未达到 2 秒目标。
  
  改进策略：
  ① 引入消息队列（Kafka/RabbitMQ）对订单创建进行异步处理，削峰填谷，
    减少同步等待时间
  ② 对订单相关数据库进行读写分离，查询走从库，减轻主库压力
  ③ 增加 Redis 缓存热点商品库存和用户信息，减少数据库访问
  ④ 对订单服务进行水平扩展，增加服务实例，配合负载均衡分散请求压力

（2）可用性（Availability）
  依据：支付网关故障导致整个购物流程不可用，说明存在单点故障和故障扩散问题。
  
  改进策略：
  ① 引入熔断机制（Sentinel），当支付网关响应超时或错误率超阈值时自动熔断，
    避免级联故障导致整个系统不可用
  ② 增加支付降级方案：支付网关故障时引导用户选择备用支付通道
  ③ 对关键服务实施冗余部署，消除单点故障

（3）安全性（Security）
  依据：发现 SQL 注入攻击日志，存在数据库安全风险。
  
  改进策略：
  ① 对所有数据库访问使用参数化查询/预编译语句，防止 SQL 注入
  ② 在 API 网关增加 WAF（Web 应用防火墙），过滤恶意请求
  ③ 对用户输入进行严格的合法性校验和过滤
  ④ 增加安全审计日志，记录异常访问行为，便于事后追溯

（4）可测试性（Testability）
  依据：运营反映新功能上线经常出现回归问题，测试覆盖率不足。
  
  改进策略：
  ① 建立完善的自动化测试体系，包括单元测试、接口测试、E2E 测试
  ② 采用 Mock 框架对外部依赖进行模拟，使各服务可以独立测试
  ③ 引入 CI/CD 流水线，每次提交自动运行测试，拦截回归问题
```

**评分要点：** 识别出 4 个质量属性各 1 分（共 4 分），每个质量属性的改进策略 1.5 分（共 6 分）。

### 问题二（8分）

请用质量属性场景描述（六要素）描述该系统的**性能质量需求**和**可用性质量需求**（每种写一个场景）。

**参考答案：**

```
性能质量属性场景：

| 场景要素       | 描述                                      |
|-------------|------------------------------------------|
| 刺激源        | 双十一促销期间的在线购物用户                    |
| 刺激          | 用户密集提交订单，并发量达到 10 万 QPS          |
| 环境          | 系统处于双十一大促峰值负载状态                   |
| 制品          | 订单服务（Order Service）                   |
| 响应          | 系统正常接受并处理订单请求，核心购物链路不超时      |
| 响应度量       | 99% 的订单请求响应时间不超过 2 秒              |


可用性质量属性场景：

| 场景要素       | 描述                                      |
|-------------|------------------------------------------|
| 刺激源        | 第三方支付网关服务                           |
| 刺激          | 支付网关因故障停止响应                        |
| 环境          | 系统正常运行中                               |
| 制品          | 支付服务及订单购物流程                         |
| 响应          | 系统检测到支付网关异常，自动熔断并切换到备用支付通道，
               不影响整体购物流程               |
| 响应度量       | 从支付网关故障到系统自动切换的时间不超过 30 秒   |
```

**评分要点：** 每个场景 4 分，六要素各 0.5 分（共 3 分），场景描述与题目背景相符（1 分）。

### 问题三（7分）

该系统中"性能"和"可用性"之间是否存在冲突？如有，请分析冲突点并给出权衡方案。

**参考答案：**

```
（1）冲突分析：

性能和可用性之间存在一定的冲突关系：

冲突点1：熔断机制 vs 性能
  为提升可用性引入熔断机制，当后端服务降级时会拒绝部分请求，
  对个别用户来说响应会变为错误，看似降低了性能（请求失败）。
  但这是以牺牲少量请求的成功率来保全整体系统的可用性。

冲突点2：同步调用 vs 异步处理
  为了保证订单数据的强一致性（可用性需求的一部分），使用同步调用链，
  但同步调用在高并发场景会导致线程阻塞，影响性能。
  而改为异步消息队列提升了性能，但可能引入数据最终一致性问题。

（2）权衡方案：

方案1（分级处理）：
  - 核心支付链路（金额强一致）采用同步调用，保证强一致性
  - 非核心操作（库存统计、消息通知）采用异步处理，提升性能
  
方案2（弹性限流）：
  - 设置合理的限流阈值，在系统过载前主动限流
  - 超过阈值时拒绝部分非核心请求，保证核心业务的响应时间

结论：对于本电商系统，建议以"可用性"为首要保障目标，在此基础上用
     分层策略兼顾性能：核心链路保可用，非核心链路保性能。
```

## 案例二：质量属性权衡 —— 安全性 vs 易用性

### 题目背景

某政务服务平台，面向普通市民提供在线办事服务，同时需要满足国家信息安全等级保护三级要求。

**业务特点：**
- 用户群体广泛，包括老年人、残障人士等特殊群体
- 涉及身份证号、住址等敏感个人信息
- 部分业务（如住房公积金查询）对安全性要求极高
- 政府要求系统对所有操作留存审计日志

**当前矛盾：**
安全团队要求：增加短信+人脸识别双重身份验证，并且每隔 30 分钟强制重新登录。
用户研究团队反映：现有安全措施已让老年用户感到操作困难，频繁重新登录影响体验。

---

### 问题四（5分）

请分析安全性和易用性之间的冲突，并给出兼顾两者的权衡架构方案。

**参考答案：**

```
（1）冲突分析：

安全性措施（双重认证+30分钟超时）的目的是保护用户敏感数据，
但带来了以下易用性问题：
- 增加了登录步骤，对老年等特殊用户操作难度大
- 频繁超时重新登录打断业务流程，用户体验差
- 可能导致用户放弃使用，反而降低平台的可用性（业务层面）

（2）权衡架构方案 —— 风险自适应认证：

核心思路：不同风险等级的操作，采用不同强度的认证方式。

低风险操作（如查询公告、下载表格）：
  → 用户名+密码即可，无需二次验证
  → 会话有效期可设为 2 小时

中风险操作（如查询个人社保信息）：
  → 首次访问需短信验证码，后续 30 分钟内免验证
  → 降低频繁验证的打扰

高风险操作（如公积金提取、修改账户绑定）：
  → 每次操作均需双重验证（短信+人脸）
  → 操作完成后该高权限自动失效

特殊用户辅助：
  → 为老年用户提供"亲属代办"功能，并有专门的辅助操作界面
  → 提供窗口服务作为线上操作的替代方案

结论：通过分级认证策略，在高敏感操作上严格保障安全性，
     在低风险操作上保证易用性，实现两者的平衡。
```
