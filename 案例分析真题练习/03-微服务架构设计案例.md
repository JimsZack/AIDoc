# 微服务架构设计案例 — 练习题

---

## 案例一：医疗信息系统微服务改造

### 题目背景

某三甲医院正在将传统单体医院信息系统（HIS）迁移为微服务架构。现有系统包含以下主要功能模块：

- **挂号预约**：患者通过 App/Web 预约挂号，支持窗口和自助机
- **门诊电子病历**：医生填写病历，诊断，开具处方
- **检验/检查**：开立检验医嘱，实验室接收和处理，结果回传
- **药房管理**：配药、发药、库存管理
- **住院管理**：床位管理、住院病历、护理记录
- **结算收费**：门诊和住院费用计算、收费、医保结算

**主要挑战：**
1. 门诊高峰期（8:00-11:00）预约挂号和收费模块压力极大，但其他模块负载正常
2. 各科室有不同的信息化需求，需要快速迭代更新，但当前单体部署导致更新需整体停机
3. 系统需要对接外部医保系统、区域健康档案平台、检验设备接口
4. 患者数据属于高度敏感数据，需满足等保三级安全要求
5. 检验结果需要在医生开单后 2 小时内回传到门诊系统，有时延要求

---

### 问题一（12分）

请对该医院信息系统进行微服务拆分设计，要求：
1. 列出至少 6 个核心微服务，说明各服务的职责和数据存储方案
2. 说明服务拆分的依据
3. 描述关键的服务间通信设计（至少举例说明 2 种通信场景）

**参考答案：**

```
（1）核心微服务拆分：

| 服务名称        | 主要职责                              | 数据存储                           |
|--------------|-------------------------------------|----------------------------------|
| 预约挂号服务    | 号源管理、在线预约、排队叫号             | MySQL（号源） + Redis（实时队列）    |
| 患者服务       | 患者信息管理、就诊记录、建档              | MySQL（患者档案）                   |
| 门诊电子病历服务 | 病历录入、诊断、处方开立                  | MySQL + 文档存储（非结构化病历）      |
| 检验检查服务    | 医嘱接收、标本管理、结果录入、结果回传       | MySQL + 消息队列（结果回传）          |
| 药房服务       | 处方审核、配药、发药、库存管理             | MySQL                            |
| 住院管理服务    | 床位管理、住院病历、护理记录、出入院         | MySQL                            |
| 收费结算服务    | 费用计算、收费、医保对接                  | MySQL（费用） + 对接医保系统          |
| 消息通知服务    | 就诊提醒、检查报告就绪通知、系统消息        | 消息队列（Kafka）                    |

（2）拆分依据：

① 按业务领域（DDD 限界上下文）：各模块对应医院不同业务流程，边界清晰
② 按访问频率差异：预约挂号和收费高峰期压力大，需要独立扩容，拆分后可单独
   水平扩展，不影响其他服务
③ 按变更频率：不同科室业务需求变化频率不同，独立服务允许按需快速迭代
④ 按数据自治：每个服务管理自己的数据，避免数据库层面的耦合

（3）服务间通信设计：

场景1（同步调用）：门诊收费 → 患者服务
  医生完成诊断、患者去收费时，收费服务需要实时查询患者身份信息和就诊记录
  用于费用计算。这是同步请求，采用 RESTful API 调用，有严格的时延要求（<500ms）。

场景2（异步消息）：检验服务 → 门诊病历服务
  检验结果产生后，检验服务向消息队列（Kafka）发布"检验结果就绪"事件，
  门诊病历服务订阅该事件后自动将结果关联到患者病历。
  使用异步消息解耦，避免同步等待，且检验服务无需关心谁消费了结果。
  这也满足了题目中"2 小时内回传"的最终一致性要求。
```

**评分要点：** 列出 6 个以上服务（6×1=6分），服务描述和存储（2分），拆分依据（2分），通信场景（2分）。

### 问题二（8分）

针对该医疗系统的**高可用**和**数据安全**需求，分别给出架构设计方案。

**参考答案：**

```
一、高可用设计：

（1）关键服务冗余部署：
  预约挂号服务、收费服务（高峰期高负载）采用多实例部署
  通过负载均衡（Nginx/K8s Service）分发请求
  配合健康检查，自动剔除故障实例

（2）数据库高可用：
  MySQL 主从复制 + 读写分离
  采用 MHA/InnoDB Cluster 实现主库故障时自动切换
  关键数据定期备份，支持时间点恢复

（3）服务熔断与降级：
  检验服务等非核心服务故障时，门诊系统可以降级处理
  （允许先保存病历，检验结果后续同步）
  使用 Sentinel 实现自动熔断，防止故障级联

（4）接诊高峰限流：
  挂号服务在高峰期设置限流策略，超出容量时排队等待而非直接报错
  通过消息队列缓冲高峰请求，削峰填谷

二、数据安全设计：

（1）传输安全：
  所有内外部接口使用 HTTPS/TLS 加密传输
  与医保系统、检验设备的对接接口使用 API 签名验证

（2）存储安全：
  患者敏感数据（身份证号、诊断信息）字段级加密存储（AES-256）
  数据库配置访问控制，不同服务使用不同的数据库账号，最小权限原则

（3）访问控制：
  统一认证中心（基于 OAuth2.0 + RBAC），不同角色（医生/护士/收费员）
  只能访问对应权限的服务和数据

（4）审计日志：
  所有涉及患者数据的读写操作记录审计日志
  审计日志不可篡改，保留至少 6 年（满足医疗行业合规要求）
```

### 问题三（5分）

请描述该系统与外部系统（医保系统、区域健康档案平台）对接时的集成架构方案。

**参考答案：**

```
（1）集成架构方案：

采用 API 网关 + 适配层 的集成模式：

外部集成架构：
  [本院系统] → [对外 API 网关] → [集成适配服务] → [外部系统]
                                                   ↳ 医保系统
                                                   ↳ 区域健康档案
                                                   ↳ 检验设备

（2）适配层设计：

医保系统对接：
  - 协议转换：医保系统采用 WebService/SOAP 协议，本院使用 REST
  - 建立医保适配服务，封装协议转换逻辑
  - 引入超时重试机制，医保系统响应慢时自动重试

区域健康档案对接：
  - 采用标准化接口（如基于 HL7 FHIR 标准）进行患者数据交换
  - 建立数据映射层，处理双方数据模型的差异

（3）接口安全保障：
  - 调用医保系统使用数字证书 + 接口签名验证
  - 对外暴露的接口限制 IP 白名单访问
  - 每次调用记录日志，便于对账和故障排查
```

## 案例二：微服务技术选型问答

### 题目背景

架构师在设计某互联网公司微服务架构时，面临多项技术选型决策。

---

### 问题四（5分）

请说明在微服务架构中，**API 网关**的主要作用，以及需要在 API 网关上实现哪些功能。

**参考答案：**

```
API 网关的主要作用：
  API 网关是微服务架构的统一入口，为客户端提供单一接入点，
  屏蔽后端服务的复杂性和分布式细节。

API 网关需要实现的核心功能：

（1）路由转发：根据请求路径和规则，将请求转发到对应的后端服务

（2）身份认证与鉴权：
  - 统一验证用户身份（JWT Token 验证）
  - 避免各服务重复实现认证逻辑

（3）限流熔断：
  - 对外部请求进行速率限制，防止后端服务被压垮
  - 后端服务异常时实施熔断，快速返回降级响应

（4）协议转换：
  - 外部 HTTP/HTTPS 转为内部 gRPC 等高效协议
  - 请求/响应格式转换（如 XML 转 JSON）

（5）日志与监控：
  - 记录所有 API 调用日志
  - 采集接口调用量、响应时间等监控指标

（6）负载均衡：将请求均衡分发到多个服务实例
```
