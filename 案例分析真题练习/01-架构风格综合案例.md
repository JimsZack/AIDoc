# 架构风格综合案例 — 练习题

---

## 案例一：在线教育平台架构设计

### 题目背景

某在线教育公司准备开发一套新的学习平台，系统需求如下：

1. **业务规模**：预计注册用户 500 万，日活用户 20 万，课程总数约 5000 门
2. **核心业务**：用户注册/登录、课程购买、视频学习、直播授课、作业批改、证书颁发
3. **关键需求**：
   - 视频点播和直播功能需要支持高并发访问
   - 各业务模块需要可以独立迭代发布，如新增"AI 辅助批改"功能不影响其他模块
   - 系统应能根据负载弹性扩容，特别是在开学季等高峰期
   - 不同模块的团队独立开发，各模块技术栈可能不同
4. **技术约束**：公司已有 Java 开发团队，计划采用云原生部署

---

### 问题一（8分）

请为该在线教育平台选择合适的架构风格，并说明选择理由（至少 4 点）。

**参考答案：**

```
（1）推荐采用微服务架构风格。

（2）选择理由如下：

① 需求匹配：系统各业务模块（用户、课程、视频、直播、作业、证书）之间功能
   边界清晰，天然适合按业务领域进行微服务拆分，每个服务独立开发、部署。

② 独立迭代需求：题目明确要求"各业务模块独立迭代发布，新增AI功能不影响
   其他模块"，而微服务架构的核心优势正是服务自治、独立发布，完全满足该需求。

③ 弹性伸缩需求：微服务架构允许针对高负载服务（如视频服务）单独进行水平
   扩展，无需扩展整个应用，配合云原生容器编排（K8s）可实现弹性伸缩。

④ 团队协作需求：题目提到"不同模块的团队独立开发、技术栈可能不同"，
   微服务架构天然支持技术异构，各服务团队可选择最适合的技术栈。

⑤ 云原生适配：微服务架构与容器化（Docker+K8s）高度契合，符合"采用云原生
   部署"的技术约束。

（3）不推荐单体架构的原因：单体架构难以支持各模块独立部署，团队协作时
    代码冲突频繁，且难以针对部分功能进行弹性扩容。
```

**评分要点：** 每个理由 1.5 分，至少答出 4 个完整理由得满分。关键词：独立部署、弹性扩容、服务自治、技术异构。

---

### 问题二（10分）

请对该系统进行微服务拆分，列出至少 5 个核心服务，并说明每个服务的主要职责和数据存储方案。

**参考答案：**

```
核心微服务拆分方案如下：

| 服务名称      | 主要职责                           | 数据存储                        |
|-------------|----------------------------------|-------------------------------|
| 用户服务      | 注册、登录、个人信息、权限管理         | MySQL（用户数据） + Redis（会话）  |
| 课程服务      | 课程信息管理、分类、搜索、推荐         | MySQL + Elasticsearch（搜索）   |
| 订单/支付服务  | 课程购买、订单管理、支付对接           | MySQL（订单数据）                 |
| 视频服务      | 视频点播、转码、上传、CDN管理          | 对象存储（OSS） + 数据库（元数据）  |
| 直播服务      | 直播课管理、推流/拉流、互动            | 对接第三方直播平台 + 数据库        |
| 作业/考试服务  | 作业发布、提交、批改、成绩管理          | MySQL                          |
| 证书服务      | 证书生成、颁发、查询验真               | MySQL + 文件存储                 |
| 消息通知服务   | 系统消息、邮件、短信推送               | 消息队列（Kafka）                 |

拆分原则：按业务领域（DDD 限界上下文）划分，每个服务拥有独立数据存储，
通过 API 或消息队列进行服务间通信。
```

**评分要点：** 列出 5 个以上合理的服务（每个 1 分），每个服务有职责说明（0.5 分），有存储方案（0.5 分）。

---

### 问题三（7分）

微服务架构引入了哪些新的挑战？针对这些挑战，应分别采取什么应对策略？（至少列出 3 个挑战和应对策略）

**参考答案：**

```
挑战1：分布式系统复杂性（服务间调用故障）
  应对策略：引入熔断机制（Hystrix/Sentinel），当下游服务异常时快速熔断，
           防止级联故障蔓延。

挑战2：数据一致性问题（分布式事务）
  应对策略：
  - 强一致场景（如支付）：使用 Seata 等分布式事务框架
  - 最终一致场景（如通知）：采用 Saga 模式或本地消息表 + 事件驱动
  
挑战3：服务发现与负载均衡
  应对策略：引入服务注册中心（Nacos/Eureka），服务实例启动时自动注册，
           调用方通过注册中心发现服务并实现客户端负载均衡。

挑战4：分布式链路追踪（问题排查困难）
  应对策略：引入链路追踪系统（SkyWalking/Zipkin），为每个请求生成唯一
           的 TraceID，串联跨服务的调用链。

挑战5：配置管理分散
  应对策略：使用统一配置中心（Nacos/Apollo），所有服务配置集中管理，
           支持配置动态刷新，无需重启服务。
```

**评分要点：** 每个完整的"挑战 + 应对策略"2 分，至少答出 3 个完整组合得满分。

## 案例二：架构风格对比分析

### 题目背景

某传统制造企业的信息化部门正在对现有 ERP 系统进行重构。当前系统是一个部署在本地服务器的单体 Java 应用，随着业务增长，出现了以下问题：

- 系统响应速度下降，高峰期经常超时
- 新功能开发周期长，上线需要整体部署
- 某个模块出错会导致整个系统不可用
- 团队扩大后，多人同时修改代码冲突严重

---

### 问题四（6分）

请对比单体架构和微服务架构在该场景下的优缺点，并说明改造方向。

**参考答案：**

```
（1）当前单体架构的问题分析：

  优点：①部署简单，环境搭建成本低；②开发初期迭代快；③无分布式复杂性

  缺点（结合题目场景）：
  ①性能瓶颈：单体无法针对热点模块单独扩容，导致高峰期整体性能下降
  ②部署耦合：新功能上线必须整体部署，发布风险大，周期长
  ③故障扩散：一个模块出错导致整体不可用，可用性差
  ④团队协作：大团队共享代码库，代码冲突严重

（2）微服务架构在该场景的优势：
  ①支持按模块独立扩容，解决性能瓶颈
  ②各服务独立部署，缩短发布周期，降低风险
  ③服务隔离，一个服务故障不影响其他服务
  ④各团队拥有独立代码仓库，减少协作冲突

（3）改造建议：
  采用渐进式改造策略，不要一次性全量重写。
  建议按优先级逐步抽取高频变更和高负载模块（如库存服务、报表服务）为独立服务，
  保留核心稳定模块在单体中，形成"扼杀者模式"的渐进演进架构。
```
