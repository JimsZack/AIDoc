# 设计模式综合案例 — 练习题

---

## 案例一：电商促销系统设计模式应用

### 题目背景

某电商平台的促销系统需要支持多种促销策略，包括：
- 满减（如满 300 减 50）
- 折扣（如打九折）
- 优惠券（固定金额抵扣）
- 积分抵现（每 100 积分抵 1 元）

当前系统使用大量 if-else 判断促销类型，代码如下（伪代码）：

```java
public double calculateDiscount(Order order, String promoType) {
    if (promoType.equals("FULL_REDUCE")) {
        // 满减逻辑...
    } else if (promoType.equals("DISCOUNT")) {
        // 折扣逻辑...
    } else if (promoType.equals("COUPON")) {
        // 优惠券逻辑...
    } else if (promoType.equals("POINTS")) {
        // 积分抵现逻辑...
    }
    // 每次新增促销类型都需要修改这里
}
```

产品经理要求：未来还会新增"买一送一""拼团优惠"等更多促销类型，希望新增时不需要修改现有代码。

---

### 问题一（8分）

1. 请指出上述代码存在的设计问题，并说明违反了哪些设计原则。
2. 推荐使用哪种设计模式进行重构，并说明理由。
3. 画出重构后的类结构（用文字描述类图结构）。

**参考答案：**

```
（1）设计问题和违反的原则：

问题1：违反**开闭原则**
  每次新增促销类型都需要修改 calculateDiscount 方法，对修改开放，不符合
  "对扩展开放、对修改关闭"的要求。

问题2：违反**单一职责原则**
  一个方法承担了所有促销类型的计算逻辑，职责过于集中。

问题3：代码难以维护和测试
  大量 if-else 嵌套，添加新类型时容易引入错误，各类型逻辑混杂，难以单独测试。

（2）推荐使用**策略模式（Strategy Pattern）**重构。

选择理由：
  ① 策略模式将每种算法（促销策略）封装为独立的类，完全解耦
  ② 新增促销类型只需新增一个策略实现类，无需修改已有代码，满足开闭原则
  ③ 每个策略类职责单一，便于独立测试
  ④ 可以在运行时动态切换促销策略，满足"用户选择不同促销方式"的需求

（3）重构后的类结构：

抽象接口：
  PromotionStrategy（接口）
    + calculateDiscount(Order order): double

具体策略类（实现 PromotionStrategy）：
  FullReduceStrategy（满减策略）
    + calculateDiscount(Order order): double
  
  DiscountStrategy（折扣策略）
    + calculateDiscount(Order order): double
  
  CouponStrategy（优惠券策略）
    + calculateDiscount(Order order): double
  
  PointsStrategy（积分抵现策略）
    + calculateDiscount(Order order): double

上下文类：
  PromotionContext（促销上下文）
    - strategy: PromotionStrategy（持有策略引用）
    + setStrategy(PromotionStrategy): void
    + executeDiscount(Order): double（委托给策略执行）

客户端：
  调用 PromotionContext.setStrategy() 选择具体策略，
  再调用 executeDiscount() 获取折扣金额。

新增"拼团优惠"只需新增 GroupBuyStrategy 类，实现 PromotionStrategy 接口，
无需修改任何现有代码。
```

**评分要点：** 设计问题（2分），设计原则（2分），推荐模式及理由（2分），类结构（2分）。

### 问题二（6分）

该促销系统还需要支持**促销活动的组合叠加**，例如"使用折扣券同时叠加满减"。请推荐合适的设计模式来支持这一需求，说明该模式的结构，并与策略模式进行对比。

**参考答案：**

```
推荐使用**装饰器模式（Decorator Pattern）**支持促销叠加。

（1）装饰器模式结构：

接口（Component）：
  PromotionCalculator
    + calculateDiscount(Order order): double

基础实现（ConcreteComponent）：
  BaseCalculator — 不叠加任何促销，返回原价

装饰器基类（Decorator，实现 PromotionCalculator）：
  AbstractPromoDecorator
    - wrapped: PromotionCalculator
    + calculateDiscount(Order): double（调用 wrapped.calculateDiscount + 自身处理）

具体装饰器（ConcreteDecorator）：
  DiscountDecorator（叠加折扣）
  FullReduceDecorator（叠加满减）
  CouponDecorator（叠加优惠券）

使用方式（伪代码）：
  PromotionCalculator calc = new BaseCalculator();
  calc = new FullReduceDecorator(calc);   // 叠加满减
  calc = new DiscountDecorator(calc);     // 再叠加折扣
  double finalPrice = calc.calculateDiscount(order);

（2）装饰器模式 vs 策略模式对比：

| 维度     | 策略模式                 | 装饰器模式                 |
|--------|------------------------|--------------------------|
| 目的     | 动态切换算法（互斥选一个）  | 动态组合增强功能（可叠加多个）|
| 结构     | Context + Strategy 接口  | Component + Decorator 包装 |
| 适用场景  | 多种等价算法，选其一执行    | 基础功能 + 多层增强的叠加    |

对于"单一促销类型"用策略模式；
对于"多种促销叠加"用装饰器模式。
两者可以结合使用。
```

## 案例二：设计模式识别与应用

### 题目背景

架构师小李负责一个企业通知中心系统的设计，系统需要：

1. 当用户账户发生以下任一事件时，自动触发通知：余额变动、登录异常、订单状态变更
2. 通知渠道支持：短信、邮件、站内信、微信推送，且可扩展
3. 系统只需要一个全局唯一的通知配置管理器，统一管理通知开关和模板

---

### 问题三（8分）

针对上述需求，请为以下三个具体设计点分别推荐合适的设计模式并说明理由：

1. 当账户事件发生时，自动触发多个通知渠道的发送
2. 支持未来新增新的通知渠道类型，无需修改现有代码
3. 全局唯一的通知配置管理器

**参考答案：**

```
需求1：账户事件触发多渠道通知 → 观察者模式（Observer Pattern）

  理由：账户服务（被观察者/Subject）状态变化时，需要自动通知多个观察者
       （各通知渠道）。这是典型的一对多通知场景，观察者模式完全匹配。

  结构：
  - Subject（AccountEventPublisher）：维护观察者列表，发布事件
  - Observer接口（NotificationHandler）：定义 onEvent(AccountEvent) 方法
  - ConcreteObserver：SmsHandler、EmailHandler、AppPushHandler、WeChatHandler
  - 新增微信推送：只需新增 WeChatHandler 实现 NotificationHandler，注册到 Subject


需求2：扩展新通知渠道无需修改现有代码 → 工厂方法模式（Factory Method Pattern）

  理由：不同通知渠道的创建逻辑不同，通过工厂方法将对象创建延迟到具体工厂子类，
       新增渠道只需新增对应的工厂子类，满足开闭原则。

  结构：
  - NotificationSender（接口）：定义 send(message) 方法
  - NotificationFactory（抽象工厂）：定义 createSender() 工厂方法
  - SmsFactory（具体工厂）：创建 SmsSender
  - EmailFactory（具体工厂）：创建 EmailSender
  - 新增渠道：新增 WeChatFactory 和 WeChatSender，无需修改现有工厂代码


需求3：全局唯一通知配置管理器 → 单例模式（Singleton Pattern）

  理由：整个系统只需要一个通知配置管理器实例，避免配置重复加载和状态不一致。
       单例模式保证类只有一个实例，并提供全局访问点。

  实现要点：
  - 私有构造函数（防止外部 new）
  - 私有静态实例引用
  - 公共静态访问方法 getInstance()
  - 线程安全：采用双重检查锁定或静态内部类实现
```

**评分要点：** 每个需求识别正确模式（1分），理由说明（1分），结构描述（0.5分），共 7.5 分（取整8分）。

---

### 问题四（3分）

以下类图片段描述的是哪种设计模式？请说明各元素对应的角色。

```
+------------------+           +------------------+
|    Component     |<----------|    Decorator     |
+------------------+           +------------------+
| + operation()   |           | - component      |
+------------------+           | + operation()   |
        ↑                       +------------------+
        |                               ↑
+------------------+           +------------------+
| ConcreteComponent |           |ConcreteDecoratorA|
+------------------+           +------------------+
| + operation()   |           | + operation()   |
+------------------+           | + addedBehavior()|
                                +------------------+
```

**参考答案：**

```
该类图描述的是**装饰器模式（Decorator Pattern）**。

各元素角色：
- Component（接口/抽象类）：定义被装饰对象和装饰器的共同接口
- ConcreteComponent（具体组件）：被装饰的原始对象，提供基础功能
- Decorator（装饰器基类）：持有 Component 的引用（组合关系），实现 Component 接口
- ConcreteDecoratorA（具体装饰器）：在调用被包装对象的 operation() 基础上，
  新增 addedBehavior() 功能，实现动态功能增强

核心特征识别：
  装饰器和组件实现相同接口 + 装饰器持有组件引用（has-a）= 装饰器模式
```
